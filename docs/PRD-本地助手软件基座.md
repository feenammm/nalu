## 1. 背景与目标

### 1.1 背景

日常工作和开发中，经常需要在多个独立工具之间频繁切换——命令行工具、文件处理脚本、API 调试器、笔记应用、剪贴板管理器、定时任务等。这些工具各自独立运行，无法共享上下文，且每个工具都有自己的启动方式和交互界面。

现有的解决方案（如 Raycast、Alfred、Obsidian）虽然提供了插件扩展能力，但存在以下问题：

- **数据归属问题**：多数产品依赖云端，个人数据不在本地掌控。
- **插件开发门槛高**：平台型产品的插件 SDK 复杂，审核流程长，个人想快速写个小工具集成进来成本过高。
- **过度设计**：很多功能面向团队或大众用户设计，对个人用户来说是负担。
- **扩展性受限**：插件之间无法互相调用，无法组合出个人工作流。

因此需要一个轻量、本地优先、插件驱动的助手基座——核心只负责 UI 框架、插件生命周期管理和命令调度，所有具体功能由插件实现，插件之间可以互相调用，形成个人工具链。

**不做会怎样**：继续在多个零散工具间切换，效率损耗持续积累，个人工作流无法沉淀。
**做了能怎样**：拥有一个完全自己掌控的本地工具基座，随时可以写一个插件把新需求集成进来，所有工具统一入口、统一交互、数据完全本地化。

### 1.2 目标用户

| 用户角色 | 特征描述 | 核心诉求 | 使用场景 |
|---------|---------|---------|---------|
| 个人用户（日常模式） | 熟悉基本桌面操作，追求效率 | 快速找到并执行需要的功能，减少工具切换 | 快速启动应用、搜索文件、执行常用操作、管理剪贴板 |
| 个人用户（开发模式） | 有编程能力，习惯命令行和脚本 | 把自己的脚本/工具封装成插件，统一管理和调用 | 封装 API 调用、自动化脚本、代码片段管理、DevOps 操作 |

### 1.3 业务目标与成功指标

| 目标 | 衡量指标 | 目标值 | 监测方式 |
|------|---------|--------|---------|
| 插件开发体验足够轻量 | 从零写一个最小插件到可运行的时间 | < 15 分钟 | 首次开发计时测试 |
| 基座启动速度满足日常使用 | 冷启动时间 | < 2 秒 | 应用启动计时 |
| 插件调用响应流畅 | 命令执行响应时间（不含插件自身逻辑） | < 200ms | 内置性能监控 |
| 个人工具链可沉淀 | 稳定运行 30 天无崩溃 | 崩溃次数 = 0 | 本地日志记录 |

## 2. 需求概述

一个本地优先的桌面助手基座，提供统一命令面板和插件生命周期管理，所有功能由可热插拔的插件实现，插件之间可互相调用。

## 3. 功能详细设计

### 3.1 命令面板（Command Palette）

**功能描述**：全局统一入口，用户通过快捷键唤起命令面板，搜索并执行已安装插件提供的命令。
**用户故事**：作为个人用户，我希望通过一个快捷键找到所有可用的操作，以便快速执行而不需要记住每个工具的入口。
**优先级**：P0

**业务规则**：
1. 全局热键唤起命令面板，默认热键为 `Cmd/Ctrl + Space`，用户可自定义。
2. 命令面板为一个顶部浮窗，包含一个搜索输入框和命令列表。
3. 搜索支持模糊匹配：输入字符与命令名称或描述进行模糊匹配（不要求完全一致）。
4. 搜索结果按以下顺序排序：最近使用过的 > 使用频率高的 > 字母序。
5. 每个命令项显示：插件图标（如有）、命令名称、命令描述、所属插件名称。
6. 按 `Enter` 执行选中命令；按 `Esc` 关闭面板；按上下箭头键切换选中项。
7. 命令面板关闭时释放焦点，不影响当前活动窗口。
8. 命令面板最多显示 20 条搜索结果，超出部分滚动查看。
9. 支持命令参数：部分命令需要输入参数，在搜索框中以 `>` 或 `:` 分隔命令名和参数（如 `translate > hello world`）。

**交互流程**：
1. 用户按下全局热键，命令面板从屏幕顶部中央弹出。
2. 焦点自动进入搜索输入框，显示最近使用的 10 条命令。
3. 用户输入关键词，列表实时过滤（输入延迟 100ms 后触发搜索）。
4. 用户通过上下箭头选择目标命令，按 Enter 执行。
5. 成功：命令执行，面板关闭（若命令为后台任务）或切换为命令输出视图（若命令有返回结果展示）。
6. 失败：面板底部显示红色 toast 提示错误信息，3 秒后自动消失。

**异常处理**：

| 异常场景 | 处理方式 | 用户提示 |
|---------|---------|---------|
| 命令执行超时（> 30 秒无响应） | 自动终止执行，释放面板 | "命令执行超时，已自动终止" |
| 插件崩溃导致命令不可用 | 标记该命令为不可用状态，灰显显示 | "插件 [插件名] 异常，请检查或重启插件" |
| 搜索无结果 | 显示空状态提示 | "未找到匹配命令，试试换个关键词？" |
| 全局热键与其他软件冲突 | 允许用户修改热键，冲突时弹出提示 | "快捷键已被占用，请在设置中修改" |

### 3.2 插件生命周期管理

**功能描述**：管理插件的安装、卸载、启用、禁用和更新。
**用户故事**：作为个人用户，我希望方便地管理我的插件集合，以便按需启用或移除功能。
**优先级**：P0

**业务规则**：
1. 插件安装方式支持两种：从本地目录加载（指定插件文件夹路径）、从 URL 安装（下载插件包到本地）。
2. 插件安装后存放在统一的本地目录 `{应用数据目录}/plugins/` 下，每个插件一个独立子目录。
3. 插件启用/禁用为即时生效，不需要重启基座。禁用后插件的命令从命令面板中隐藏，但插件数据保留。
4. 插件卸载时，弹出确认对话框。确认卸载后删除插件目录和关联的配置数据。插件运行产生的用户数据（如有）单独存放在 `{应用数据目录}/data/{插件ID}/`，卸载时提示用户是否保留。
5. 插件更新：基座启动时检查已安装插件的更新（通过插件 manifest 中声明的更新源）。有更新时在插件管理页面显示更新标记，用户手动确认更新。
6. 插件之间可声明依赖关系：在 manifest 中声明 `dependencies` 字段。安装时若依赖不满足，提示用户先安装依赖插件。
7. 每个插件必须包含一个 `manifest.json` 文件，声明以下字段：

| 字段 | 必填 | 说明 |
|------|------|------|
| `id` | 是 | 插件唯一标识符，格式 `author.plugin-name` |
| `name` | 是 | 插件显示名称 |
| `version` | 是 | 语义化版本号 |
| `description` | 是 | 一句话描述插件功能 |
| `author` | 是 | 作者名 |
| `entry` | 是 | 入口文件相对路径 |
| `commands` | 是 | 注册的命令列表 |
| `dependencies` | 否 | 依赖的其他插件 ID 列表 |
| `permissions` | 否 | 需要的系统权限（文件系统、网络、剪贴板） |
| `updateUrl` | 否 | 插件更新检查地址 |

**交互流程**：
1. 用户从设置页面进入「插件管理」。
2. 页面显示已安装插件列表，每个插件卡片包含：图标、名称、版本、描述、启用/禁用开关、卸载按钮。
3. 安装插件：点击页面顶部「安装插件」按钮，弹出对话框提供两种安装方式（选择本地目录 / 输入 URL），选择后执行安装。
4. 成功安装：插件出现在列表中，命令面板立即可用。
5. 卸载插件：点击卸载按钮，弹出确认框，确认后执行卸载。

**异常处理**：

| 异常场景 | 处理方式 | 用户提示 |
|---------|---------|---------|
| manifest.json 格式不合法 | 安装失败，显示具体错误字段 | "插件配置无效：缺少必填字段 [字段名]" |
| 依赖插件未安装 | 安装时提示，允许强制安装（不推荐） | "该插件依赖 [插件A]、[插件B]，是否一并安装？" |
| 插件目录损坏 | 启动时检测并标记为异常 | "插件 [插件名] 文件损坏，请重新安装" |
| 下载插件时网络中断 | 清理已下载的不完整文件，提示重试 | "下载中断，请检查网络后重试" |

### 3.3 插件间通信与组合

**功能描述**：插件之间可以互相调用命令和共享数据，支持将多个命令组合为工作流。
**用户故事**：作为个人用户，我希望把多个插件的功能串联起来形成自动化流程，以便一次操作完成复杂任务。
**优先级**：P0

**业务规则**：
1. 每个插件通过基座提供的 API 注册自己的命令，其他插件可通过命令 ID 调用。
2. 命令调用的数据格式统一为 JSON：`{ commandId: string, params: object, caller: string }`。
3. 命令返回结果统一为 JSON：`{ success: boolean, data: any, error: string | null }`。
4. 工作流定义：用户可在基座中将多个命令串联为一个工作流（Workflow），前一个命令的输出作为后一个命令的输入。
5. 工作流支持条件分支：根据上一步返回值决定下一步执行哪个命令。
6. 工作流本身保存为一个特殊的"工作流插件"，可在命令面板中直接调用。
7. 插件间通信超时默认 10 秒，可在基座设置中调整。

**交互流程**：
1. 用户进入「工作流编辑器」页面（通过命令面板搜索 "工作流" 进入）。
2. 点击「新建工作流」，输入工作流名称和描述。
3. 在可视化画布上，从左侧命令列表拖拽命令节点到画布。
4. 用连线将命令节点串联，连线上可设置条件（等于/包含/大于等）。
5. 为每个命令节点配置输入参数（可引用上游节点的输出）。
6. 点击「保存」，工作流出现在命令面板中可调用。

**异常处理**：

| 异常场景 | 处理方式 | 用户提示 |
|---------|---------|---------|
| 工作流中间步骤执行失败 | 暂停工作流，高亮失败节点，显示错误信息 | "步骤 [步骤名] 执行失败：[错误信息]。是否跳过继续？" |
| 被调用的插件未安装 | 工作流编辑时校验依赖，阻止保存 | "工作流依赖的插件 [插件名] 未安装" |
| 循环引用检测 | 保存时校验，发现循环则阻止 | "检测到循环依赖：A → B → A，请修改流程" |

### 3.4 插件开发 SDK

**功能描述**：提供插件开发工具包，包括项目模板、API 文档和开发调试工具。
**用户故事**：作为个人用户（开发模式），我希望快速创建一个新插件，以便把自己的脚本或工具集成到基座中。
**优先级**：P0

**业务规则**：
1. 提供命令行工具 `pluginhost init`，交互式生成插件项目模板，包含 manifest.json、入口文件、示例命令。
2. 插件运行时支持两种模式：进程内插件（与基座同进程，适合轻量级插件）、独立进程插件（单独进程运行，适合计算密集型或需要隔离的插件）。
3. 基座向插件暴露标准 API，包括：注册命令、读写配置、文件系统访问、HTTP 请求、剪贴板读写、通知推送、日志记录。
4. 插件开发模式下，支持热重载：修改插件代码后自动重新加载，无需手动重启。
5. 提供 `pluginhost dev` 命令启动开发模式，自动监听插件目录变化并热重载。
6. 提供 `pluginhost test` 命令运行插件自带的测试用例。
7. API 文档以本地 HTML 形式内嵌在基座中，无需联网即可查看。

**交互流程**：
1. 用户在终端执行 `pluginhost init`。
2. CLI 交互式询问：插件名称、描述、作者、插件模式（进程内/独立进程）。
3. 生成插件项目目录，包含完整的脚手架代码。
4. 用户编写插件逻辑后，执行 `pluginhost dev` 进入开发模式。
5. 基座自动加载该插件，代码修改后自动重载。
6. 在命令面板中可立即测试新注册的命令。

**异常处理**：

| 异常场景 | 处理方式 | 用户提示 |
|---------|---------|---------|
| 插件代码语法错误 | 热重载失败，终端显示编译错误 | "插件加载失败：[文件名] 第 [行号] 行语法错误" |
| 插件 API 调用越权 | 拦截调用，记录日志 | "插件 [插件名] 尝试访问未授权的 [权限类型]，请在 manifest 中声明 permissions" |
| 插件内存泄漏（超过 500MB） | 发出警告，用户可选择重启插件 | "插件 [插件名] 内存占用过高（[具体数值]），建议重启" |

### 3.5 设置中心

**功能描述**：统一管理基座全局设置和各插件的配置项。
**用户故事**：作为个人用户，我希望在一个地方管理所有设置，以便快速调整行为偏好。
**优先级**：P1

**业务规则**：
1. 设置中心分为两个区域：「基座设置」和「插件设置」。
2. 基座设置包括：全局热键、主题（亮色/暗色/跟随系统）、启动行为（开机自启/最小化到托盘）、数据存储路径、性能选项。
3. 插件设置：每个插件可通过 manifest 声明配置项 schema（字段名、类型、默认值、描述），基座自动渲染对应的设置表单。
4. 支持的配置项类型：文本输入（string）、数字输入（number）、开关（boolean）、下拉选择（enum）、颜色选择（color）、快捷键录入（hotkey）、文件/目录选择（path）。
5. 所有设置实时保存到本地配置文件 `{应用数据目录}/config.json`，修改后立即生效。
6. 支持配置导入/导出：导出为 JSON 文件，导入时覆盖当前配置（需二次确认）。

**交互流程**：
1. 用户通过命令面板搜索 "设置" 或点击托盘图标菜单进入设置中心。
2. 左侧导航栏显示「基座设置」和已安装插件列表（点击可进入对应插件设置）。
3. 右侧展示对应设置表单。
4. 修改配置项后即时生效，无需保存按钮。
5. 导出配置：点击右上角「导出配置」按钮，选择保存路径。
6. 导入配置：点击「导入配置」按钮，选择 JSON 文件，弹出确认框后覆盖。

**异常处理**：

| 异常场景 | 处理方式 | 用户提示 |
|---------|---------|---------|
| 配置文件损坏 | 自动从备份恢复（基座每次启动时创建配置快照） | "配置文件异常，已从备份恢复" |
| 导入的配置格式不兼容 | 校验失败，拒绝导入 | "配置文件格式不匹配，请检查文件" |
| 插件声明的配置 schema 无效 | 该插件设置页显示错误提示 | "插件 [插件名] 的配置定义有误，请联系插件作者" |

### 3.6 系统托盘与状态栏

**功能描述**：基座在系统托盘区常驻运行，提供快捷菜单和状态指示。
**用户故事**：作为个人用户，我希望基座在后台安静运行并能快速访问，以便随时使用而不占用任务栏空间。
**优先级**：P1

**业务规则**：
1. 基座启动后最小化到系统托盘，不显示主窗口。
2. 托盘图标右键菜单包含：打开命令面板、打开设置、最近使用的 5 个命令（快捷执行）、退出。
3. 托盘图标状态指示：正常运行（默认图标）、有插件异常（图标加黄色角标）、正在更新（图标加旋转动画）。
4. 左键点击托盘图标：打开命令面板。
5. 退出操作需二次确认，防止误操作。

**交互流程**：
1. 基座启动，托盘区出现图标。
2. 用户右键点击图标，弹出菜单。
3. 选择菜单项执行对应操作。
4. 选择「退出」时弹出确认对话框。

**异常处理**：

| 异常场景 | 处理方式 | 用户提示 |
|---------|---------|---------|
| 基座主进程崩溃 | 操作系统层面自动重启（配置守护进程） | 重启后托盘提示 "应用已自动恢复" |
| 托盘图标被系统隐藏 | 在设置中提醒用户将图标固定到托盘可见区域 | "建议在系统设置中将 PluginHost 图标设为始终显示" |

### 3.7 数据管理与备份

**功能描述**：统一管理基座和各插件的数据存储，提供备份与恢复能力。
**用户故事**：作为个人用户，我希望我的数据全部在本地且有备份机制，以便出问题时可以恢复。
**优先级**：P1

**业务规则**：
1. 数据存储目录结构：`{应用数据目录}/` 下分为 `plugins/`（插件代码）、`data/`（插件运行数据）、`config.json`（全局配置）、`workflows/`（工作流定义）。
2. 每个插件的运行数据存放在独立目录 `data/{插件ID}/`，插件只能读写自己的数据目录。
3. 手动备份：在设置中心提供「备份所有数据」按钮，将所有数据打包为 `.tar.gz` 文件。
4. 自动备份：每天首次启动时自动创建增量备份，保留最近 7 天的备份。
5. 恢复：从备份文件恢复，恢复前自动备份当前数据（防止恢复后又想回退）。
6. 备份文件存放位置：`{应用数据目录}/backups/`，用户可自定义。

**交互流程**：
1. 用户在设置中心「数据管理」页面查看存储空间使用情况和备份历史。
2. 手动备份：点击「立即备份」，显示进度条，完成后显示备份文件路径。
3. 恢复：选择备份记录中的某一条，点击「恢复」，弹出确认框（告知当前数据会先备份），确认后执行。

**异常处理**：

| 异常场景 | 处理方式 | 用户提示 |
|---------|---------|---------|
| 磁盘空间不足导致备份失败 | 检测可用空间，不足时提醒清理 | "磁盘空间不足，请清理后重试（需至少 [数值] MB）" |
| 备份文件损坏 | 校验备份完整性（SHA-256），损坏则标记不可用 | "该备份文件已损坏，无法恢复" |
| 插件正在运行时尝试恢复 | 暂停相关插件后执行恢复 | "恢复过程中需要暂时停用插件，是否继续？" |

### 3.8 日志与诊断

**功能描述**：记录基座和插件的运行日志，提供诊断工具帮助排查问题。
**用户故事**：作为个人用户（开发模式），我希望能查看运行日志，以便排查插件异常问题。
**优先级**：P2

**业务规则**：
1. 日志分级：DEBUG、INFO、WARN、ERROR，默认记录 INFO 及以上级别。
2. 日志存储：本地文件 `{应用数据目录}/logs/`，按日期分文件，自动清理 30 天前的日志。
3. 提供日志查看器页面：支持按级别过滤、按关键词搜索、按插件过滤。
4. 诊断工具：一键收集当前环境信息（系统版本、基座版本、已安装插件列表及版本、内存占用）生成诊断报告文件。

**交互流程**：
1. 用户通过命令面板搜索 "日志" 进入日志查看器。
2. 默认显示最近 100 条日志，可按条件筛选。
3. 搜索 "诊断" 生成诊断报告。

**异常处理**：

| 异常场景 | 处理方式 | 用户提示 |
|---------|---------|---------|
| 日志文件过大（> 100MB） | 自动轮转，当前文件截断并归档 | 无需提示，自动处理 |
| 日志写入失败 | 降级为内存缓存，恢复写入后刷盘 | "日志写入异常，已暂存到内存" |

## 4. 非功能需求

| 类别 | 要求 | 验收标准 |
|------|------|---------|
| 性能 | 基座冷启动时间 | < 2 秒（不含插件加载），插件延迟加载 |
| 性能 | 命令面板搜索响应 | < 100ms（100 个命令以内） |
| 性能 | 内存占用（基座自身） | < 100MB（不含插件） |
| 安全 | 插件权限隔离 | 插件只能访问 manifest 中声明的权限，越权调用被拦截并记录日志 |
| 安全 | 数据本地化 | 所有数据存储在本地，基座不主动发起任何网络请求（插件自身请求除外） |
| 兼容性 | 操作系统支持 | macOS 12+、Windows 10+（优先支持 macOS） |
| 稳定性 | 插件崩溃隔离 | 独立进程插件崩溃不影响基座和其他插件 |
| 可访问性 | 键盘操作 | 所有核心操作均可通过键盘完成 |

## 6. 验收标准

| 编号 | 场景 | Given（前提） | When（操作） | Then（预期结果） |
|------|------|-------------|-------------|-----------------|
| AC-01 | 唤起命令面板 | 基座已启动并在后台运行 | 按下全局热键 Cmd+Space | 命令面板在屏幕顶部中央弹出，焦点在搜索框，显示最近使用的命令 |
| AC-02 | 搜索命令 | 已安装 3 个插件共 15 个命令 | 在搜索框输入关键词 "file" | 100ms 内显示匹配结果，按使用频率排序 |
| AC-03 | 执行命令 | 命令面板已打开且搜索结果已显示 | 用方向键选择命令后按 Enter | 命令执行，面板关闭（后台命令）或切换到输出视图 |
| AC-04 | 安装本地插件 | 本地存在合法的插件目录（含有效 manifest.json） | 在插件管理页选择「从本地目录安装」并选择该目录 | 插件安装成功，出现在插件列表中，命令面板可搜索到新命令 |
| AC-05 | 安装非法插件 | 本地目录中缺少 manifest.json | 尝试安装该目录 | 安装失败，提示 "插件配置无效：缺少 manifest.json" |
| AC-06 | 禁用插件 | 插件 A 已安装并启用 | 在插件管理页关闭插件 A 的开关 | 插件 A 的命令从命令面板消失，插件数据保留 |
| AC-07 | 卸载插件 | 插件 A 已安装 | 点击卸载按钮并确认 | 插件目录和配置被删除，提示是否保留用户数据 |
| AC-08 | 插件间调用 | 插件 A 注册了命令 `a.hello`，插件 B 通过 API 调用该命令 | 插件 B 执行调用 `a.hello` | 收到返回结果 `{ success: true, data: ... }` |
| AC-09 | 工作流执行 | 已创建包含 3 个步骤的工作流 | 通过命令面板执行该工作流 | 3 个步骤依次执行，最终输出结果 |
| AC-10 | 工作流中间步骤失败 | 工作流第 2 步会报错 | 执行该工作流 | 第 1 步成功，第 2 步失败时暂停，显示错误信息，询问是否跳过继续 |
| AC-11 | 插件开发热重载 | 使用 `pluginhost dev` 进入开发模式 | 修改插件入口文件并保存 | 插件自动重新加载，终端显示 "插件 [名称] 已重载" |
| AC-12 | 插件权限拦截 | 插件 manifest 未声明 `filesystem` 权限 | 插件代码尝试读取本地文件 | 调用被拦截，日志记录越权访问，插件收到权限错误 |
| AC-13 | 修改全局热键 | 基座运行中 | 在设置中心将全局热键改为 Cmd+Shift+P | 新热键立即生效，旧热键不再能唤起面板 |
| AC-14 | 配置导出 | 基座有自定义设置和多个插件配置 | 点击「导出配置」 | 生成 JSON 文件，包含所有基座和插件配置 |
| AC-15 | 配置导入 | 有之前导出的配置 JSON 文件 | 选择文件导入并确认 | 当前配置被覆盖，二次确认前显示 "将覆盖当前所有设置" |
| AC-16 | 自动备份 | 今天首次启动基座 | 基座正常启动 | 自动创建增量备份到 backups 目录 |
| AC-17 | 手动备份恢复 | 存在之前的备份文件 | 选择备份并确认恢复 | 恢复前自动备份当前数据，然后从选定备份恢复 |
| AC-18 | 托盘右键菜单 | 基座在托盘运行 | 右键点击托盘图标 | 显示菜单：打开命令面板、打开设置、最近 5 个命令、退出 |
| AC-19 | 冷启动性能 | 安装 10 个插件 | 冷启动基座 | 主进程启动 < 2 秒，插件延迟加载，命令面板可用 < 3 秒 |
| AC-20 | 插件崩溃隔离 | 独立进程插件发生未捕获异常崩溃 | 插件崩溃 | 基座和其他插件不受影响，托盘图标显示黄色角标，提示插件异常 |

## 8. 风险与依赖

| 风险 | 概率 | 影响 | 缓解措施 |

| 插件 SDK API 设计反复 | 高 | 中 — 影响开发效率和已有插件兼容性 | 早期用 2-3 个真实插件场景验证 API 设计，API 版本化管理，承诺向后兼容 |
| 个人项目时间投入不稳定 | 高 | 中 — 影响整体排期 | 严格按 MVP 范围执行，先做出可用的最小版本，避免过度设计 |
| 插件间通信的数据格式变更 | 低 | 中 — 影响已有插件和工作流 | 通信协议从 v1 开始定义，版本化，向后兼容 |
